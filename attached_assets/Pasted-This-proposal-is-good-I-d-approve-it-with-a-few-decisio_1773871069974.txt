This proposal is good. I’d approve it with a few decisions and one adjustment.

The overall structure is right:
	•	single versioned JSON backup
	•	raw manual data as source of truth
	•	recompute derived fields on import
	•	exclude recommendations
	•	migration pipeline
	•	replace/merge options
	•	no change to current app entry flows

That is the correct preservation architecture.

My decisions:

1. Merge default
Default to Replace all for restore from backup, but keep Merge as an advanced secondary option.

Reason:
Your stated fear is disaster recovery after republish/reset. In that situation, the safest mental model is:

“this backup is my truth; restore me exactly.”

Merge is useful, but it is more ambiguous and easier to misunderstand. So:
	•	primary button: Restore Backup (Replace All)
	•	secondary button: Merge Backup Into Existing Data

And yes, require the explicit typed confirmation for replace.

2. Entry point
Do not add a new bottom tab.

Put it in a small Data / Backup screen reachable from a settings-style icon or overflow/menu entry, ideally from an existing place like Log or a lightweight Settings screen.

Reason:
Backup/restore is critical, but not a daily workflow. A bottom tab would be too prominent.

3. CSV bundle
Do JSON-only for this build.
CSV can come later.

Reason:
JSON is the canonical restore format, and you’re close to publish. CSV is nice for analysis, but it adds extra surface area and complexity. Get the preservation layer correct first.

The one adjustment I want:

Add app build metadata to _meta
Not just:
	•	schemaVersion

Also include something like:
	•	appVersion
	•	backupFormatVersion if separate from schema
	•	maybe deviceExportedFrom if easy, but optional

So _meta becomes more like:
	•	schemaVersion
	•	appId
	•	appVersion
	•	exportedAt
	•	recordCounts

That will help future debugging.

Two more recommendations I’d give Replit before build:

First, on import preview, also show:
	•	earliest record date
	•	latest record date

That will help confirm the backup is the right one.

Second, for emotional safety, add a very clear success message after export:
	•	file created
	•	timestamp
	•	record counts
	•	whether share/save completed
